# Solar Panel Anomaly Detection - Raspberry Pi Client Suite

This directory contains the edge application suite designed to run on a local Raspberry Pi edge computer. The client periodically checks device status, updates a local telemetry history store, and pulls the latest neural classifier checkpoints from the central web app.

---

## 📂 Deployment Package Structure

Ensure you have the following folder layout on the edge device:

```text
RaspberryPi/
│
├── requirements.txt         # Package dependencies (PyTorch CPU, Pandas, etc.)
├── daily_check.py           # Local health check CLI program & importable library
├── sync_model.py            # Urllib-based automatic model synchronizer client
├── scheduler.py             # Background daemon orchestrating check/sync loops
│
├── test_daily_check.py      # Simulator to play CSV telemetry day-by-day
├── test_sync_model.py       # Simulator to test version checks and weights sync
│
├── device_history.db        # SQLite Local Database (Stores telemetry; autogenerated)
├── model_version.txt        # Local text file tracking current model version index
│
└── models/                  # Neural network configs & checkpoints (Auto-synced)
    ├── lstm_fault_detector.pth
    ├── model_config.pkl
    ├── scaler.pkl
    ├── label_encoder.pkl
    └── feature_columns.pkl
```

> [!IMPORTANT]
> **Initial Setup**: Before running your first check, copy the contents of `PipeLine/models/` into the `RaspberryPi/models/` folder. Alternatively, configure the model synchronizer (`sync_model.py`) to download them from your server.

## ⚙️ Environment Setup on Raspberry Pi

### 1. Create Virtual Environment
SSH into your Raspberry Pi, navigate to the deployment folder, and run:
```bash
cd RaspberryPi
python3 -m venv .venv
source .venv/bin/activate
```

### 2. Install Dependencies
Install packages inside the virtual environment:
```bash
pip3 install -r requirements.txt --extra-index-url https://download.pytorch.org/whl/cpu
```

> [!TIP]
> On ARM-based edge systems (like Raspberry Pi OS), building heavy wheels like `torch` or `scikit-learn` from source is extremely resource-intensive. If pip installation times out or crashes:
> - Use the precompiled CPU wheel index as configured above.
> - Alternatively, install packages globally via system package manager:
>   `sudo apt install python3-torch python3-pandas python3-numpy python3-sklearn`

## 🚀 Edge Applications Guide

### 1. Daily Health Check (`daily_check.py`)
Evaluates telemetry logs for a specific ESP32 datalogger device.

* **Database Logging**: The script inserts the input day's data into the local `device_history.db` database.
* **Sequence Alignment**: It queries the database for the last 30 days of data for that device's serial number. If less than 30 records exist, it pads the sequence with zero-rows to satisfy the PyTorch LSTM input dimensions.
* **Command Line Arguments**:
  * `daily_logs`: (Required) Path to daily parsed JSON file or raw JSON string.
  * `serial_number`: (Required) String representing the unique serial number of the logger.
  * `--db-path`: Custom path to the SQLite history database.
  * `--model-dir`: Custom path to the folder containing model files.
  * `--clear-history`: Reset all historical database entries for this serial number prior to evaluation.
  * `--json`: Print machine-readable JSON output instead of the formatted console report.
  * `--email-alert`: Enable SMTP notifications on anomaly detection (also configurable via `EMAIL_ALERT` environment variable).
  * `--smtp-server`: SMTP server host (defaults to `smtp.gmail.com` or `SMTP_SERVER` env).
  * `--smtp-port`: SMTP server port (defaults to `587` or `SMTP_PORT` env).
  * `--smtp-user`: SMTP authentication username (defaults to `SMTP_USER` env).
  * `--smtp-password`: SMTP authentication password (defaults to `SMTP_PASSWORD` env).
  * `--smtp-sender`: Sender email address (defaults to `SMTP_SENDER` env).
  * `--email-recipient`: Recipient email address (defaults to `EMAIL_RECIPIENT` env).
  > [!TIP]
  > When run via `scheduler.py` in `debugging` mode or within the `test_daily_check.py` playback simulator, the system activates email alert suppression, ensuring only a **single alert email is sent per session** to avoid flooding your inbox.

#### Standalone Execution:
```bash
python3 daily_check.py parsed_log_today.json ESP32_SN_98765
```

#### Library Integration (Python):
```python
from daily_check import run_daily_check

result = run_daily_check(
    daily_data={"vbat_min_v": 12.5, "vbat_max_v": 13.8, "ah_charge_ah": 19.0, "ah_load_ah": 14.5, "vpv_max_v": 18.0, "ipv_max_a": 3.0, "soc_pct": 85.0, "temp_max_c": 30.0, "temp_min_c": 20.0, "night_h": 10.0},
    serial_number="ESP32_SN_98765"
)
print("Classification:", result["prediction"]["anomaly_label"])
print("Confidence:", result["prediction"]["confidence"])
```

### 2. Model Synchronizer (`sync_model.py`)
Downloads newer neural network weights from the central web server.

* **Network Polling**: Queries `/api/model/version` to get the latest version integer.
* **Atomic Save**: If the server version is greater than the version in `model_version.txt`, it downloads the new `lstm_fault_detector.pth` from `/api/model/download` into a temporary file, verifies it, and replaces the local copy atomically.
* **Command Line Arguments**:
  * `--server-url`: (Required) Server URL base (e.g. `http://192.168.1.100:5000`).
  * `--model-dir`: Path to the local models folder.
  * `--timeout`: Connection timeout limit (default: 15s).
  * `--json`: Print machine-readable confirmation JSON.

#### Standalone Execution:
```bash
python3 sync_model.py --server-url http://192.168.1.100:5000
```

#### Library Integration (Python):
```python
from sync_model import sync_if_needed

updated = sync_if_needed(
    server_url="http://192.168.1.100:5000",
    model_dir="./models"
)
if updated:
    print("New model weights loaded. Reloading PyTorch model...")
```

---

### 3. Orchestration Daemon (`scheduler.py`)
Runs in the background to automate periodic tasks.

* **Modes**:
  * `debugging`: Polling checks for server updates occur every 2 minutes, and telemetry checks run every 1 minute using sample variables.
  * `production`: Polling checks occur weekly (every 7 days), and daily telemetry checks run every 24 hours.
* **Execution**:
  ```bash
  python3 scheduler.py --mode production --server-url http://192.168.1.100:5000
  ```

## 🧪 Simulation & Test Scenarios

Verify client behaviors without remote deployment:

### 1. Telemetry Simulation (`test_daily_check.py`)
Simulates multi-day edge operation. Reads chronological rows from a telemetry CSV file and feeds them to the daily check pipeline day-by-day.
* **Command**:
  ```bash
  python3 test_daily_check.py --csv solar_telemetry.csv --delay 1.5 --serial ESP32_SIM_01 --clear-history
  ```

### 2. Network Synchronizer Simulation (`test_sync_model.py`)
Runs mock server requests in sequence. Patches request sockets to simulate version increments and verify download replacement operations locally.
* **Command**:
  ```bash
  python3 test_sync_model.py --attempts 5 --delay 2 --simulate-updates
  ```
